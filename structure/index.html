1. Фундамент — 6 файлов в корне

База системы. Меняются редко, но влияют на каждое решение. Это «правда о себе» — на чём всё стоит.

  • CLAUDE.md Главная конституция. Читается перед каждым моим сообщением. Кто я, три цели 90 дней, правила контекста, что НЕ делать. Это роутер, а не контент.
  • me.md Кто я, ценности, антицели, главная цель, фильтры решений. Стержень личности.
  • money.md Cashflow, runway, долги, цели по горизонтам (6/12 мес, 10 лет), фонды. Личные финансы, не финансы клиентов.
  • family.md Семейный снимок: миграция, дети, документы, договорённости с Викой. Не работать в выходные — отсюда.
  • decisions.md Журнал стратегических решений с YAML-тегами scope (strategic / finance / family / fishok / octo / pipeline). Append-only.
  • this-week.md Недельный план: P1/P2/P3 задачи, блоки по проектам, гипотеза недели. Обновляется в /weekly-dialogue.

2. Циклы — где живёт время

Три такта: день, неделя, по запросу. Через них стекает сырьё, генерируются своды и обновляется state.

📥 inbox/ — приёмник сырья дня

  • inbox/today.md Активный накопитель сырья за сегодня. /intake пишет сюда. Читается в /day-close вечером. В обычном режиме НЕ читается.
  • inbox/_processed/ Архив старых inbox по датам. Создаётся /day-close при закрытии дня. YYYY-MM-DD.md формат.
  • inbox/README.md Краткая инструкция к корневому inbox.

📅 daily/ — своды дней

Файлы YYYY-MM-DD.md. Генерируются автоматически в /day-close. пока пусто — первые своды появятся при первом запуске day-close.

📊 weekly/ — рефлексии-диалоги

Файлы YYYY-Wnn.md. Генерируются в /weekly-dialogue. пока пусто

3. Клиенты — изолированные проекты

Каждый клиент — отдельная папка с собственной структурой. Никакого «общего» state — изоляция предотвращает утечки контекста и экономит токены.

🐙 clients/octo/ активный детально

Octo Relocate — релокационно-визовая компания. Главный путь к equity-портфелю через канал Сергея. Самый детально проработанный клиент.

4. Практика — методология Operational Foundation

Моя методология работы с клиентами. Это «продукт» — то, что я продаю стартапам.

  • practice/operational-foundation.md Главный документ методологии: позиционирование, ICP, продуктовая лестница (4 уровня), процесс Discover → Architect → Build → Handover, 6 модулей Build.
  • practice/pricing-ladder.md Продуктовая лестница: L1 Diagnostic (€3-5К) → L2 Foundation (€8-15К + 1-2% equity) → L3 Full OS (€18-30К + 2-3% equity) → L4 Retainer (€2-4К/мес).
  • practice/cases.md Упакованные кейсы: Silver Spa, Fishok, Octo — для портфолио и для продажи методологии.
  • practice/templates/ Переиспользуемые шаблоны: discovery-интервью, регламенты, отчёты. README.md внутри.

5. AI-команда — .claude/

Конфигурация AI-инструментов: субагенты с изолированным контекстом, скиллы как атомарные процедуры, hooks для автоматизации.

🧑‍💼 .claude/agents/ — 4 субагента

  • business-operator.md Операционный архитектор. Классифицирует inbox, обновляет state-файлы клиентов, помогает с регламентами. Используется в /day-close.
  • finance-director.md Семейный и проектный финдиректор. Cashflow, runway, долги, equity-портфель. Подключается в /day-close при денежных событиях.
  • context-architect.md Архитектор контекста AndreiOS. Чистит дубли, ищет противоречия, предлагает структурные изменения. Запускается вручную ~раз в 2-4 недели.
  • octo-advisor.md Внутренний советник проекта Octo. Архетип: фаундер mobility-индустрии + AI engineer. Полная изоляция от других клиентов. только Octo

🛠 .claude/skills/ — 5 скиллов

  • intake/ /i, /intake — микро-операция: добавить сырую запись в inbox/today.md. Десятки раз в день.
  • day-close/ /day-close — главный регулярный цикл. Разбирает inbox, классифицирует, обновляет state, пишет свод. Шаг 2a проверяет проектные inbox-ы. 1 раз в день вечером.
  • weekly-dialogue/ /weekly-dialogue, /weekly — недельная рефлексия в формате диалога 60-90 мин. Не отчёт. Пятница вечером.
  • update-state/ /update-state <путь> — точечное обновление state.md проекта вне регулярного цикла. Редко, по запросу.
  • octo-intake/ /octo-intake — разбор сырья из clients/octo/inbox/ с классификацией по правилам Octo. только Octo

⚙️ .claude/settings.json

Hooks (Stop hook → авто-коммит сессии) + permissions (allow/deny список команд). Подробности на toolbox →

6. Архив и legacy

📦 archive/

Старые версии файлов, исторические артефакты. Никогда не читать без явного запроса (правило корневого CLAUDE.md).

  • archive/_gd-старые-версии/ Распиленные старые контексты прошлой системы: Контекст_GD_май_2026.md, Резюме_Octo.md. Перенесены сюда после переработки в новую структуру.

📁 Gd контекст/ legacy

Папка с историческими файлами прошлой версии системы. Постепенно перерабатывается: значимое — в новую структуру, остальное — в архив.

  • Operational_Foundation_v1.md Старая версия методологии. Актуальная — в practice/operational-foundation.md.
  • Лог_стратегических_решений.md Старый лог решений. Актуальный — корневой decisions.md.
  • Резюме_Fishok.md Резюме кейса Fishok. Актуальное — будет распилено в clients/fishok/ по мере необходимости.
  • США.md, История проекта "Испания".md Стратегические заметки. Актуальная семейная картина — корневой family.md.

📁 GD/ legacy

Папка эксперимента _NEW_SYSTEM/ прошлой версии — отложенная система 6 GPT-агентов. На паузе до стабилизации дохода €8К+/мес.

Принцип миграции: старые папки Gd контекст/ и GD/ намеренно не трогаем массово. Распиливаем только то, что нужно для текущей цели (так как сделали с Octo). Остальное — естественно отмирает или мигрирует когда становится нужным.

7. Принципы, держащие систему

Правило роста: новая папка/файл — только когда (а) действующий перерос 200 строк, (б) появилось ≥3 однотипных артефакта, или (в) явное решение «нужно для одной из трёх целей 90 дней». Никогда «на всякий случай».
Изоляция контекста: при работе с проектом — только его файлы. octo-advisor не видит fishok/. Основной Claude в общей сессии не лезет в архив. Это экономит токены и предотвращает утечки между клиентами.
Append-only для decisions: ADR никогда не редактируются. Если решение изменилось — пишется новый ADR со ссылкой на старый. Это сохраняет историю мышления.
Kill-switch: если 14 дней нет коммитов в репозитории — система признана мёртвой. 30 дней не строим следующую итерацию. Ретроспектива «почему не работало» прежде чем продолжать.

Карта обновляется по мере роста системы. Если что-то добавилось / изменилось / удалено — здесь должно отразиться.
