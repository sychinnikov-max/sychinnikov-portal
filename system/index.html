AndreiOS — карта системы

Как устроена моя операционная система: фундамент, клиенты, AI-команда, циклы работы и правила.

Главный принцип: AndreiOS - это не «инструмент управления задачами». Это контекстная база, в которую раз в день стекаются события, а AI-команда помогает разложить их по проектам, обновить состояние и поставить приоритеты на завтра.

Чтение схемы: Андрей кидает сырьё в inbox через день, вечером /day-close разбирает его и обновляет нужные файлы. Раз в неделю /weekly-dialogue переосмысляет состояние и обновляет план.

2. Фундамент — 5 файлов в корне

Это база. Меняется редко, но влияет на каждое решение.

  • me.md Кто я, ценности, антицели, главная цель, двухтактная стратегия, фильтры решений
  • money.md Cashflow, runway, долги, цели по горизонтам (6/12 мес, 10 лет), фонды
  • family.md Семейный снимок: миграция, дети, документы, договорённости с Викой
  • decisions.md Журнал стратегических решений с YAML-тегами scope (strategic/finance/family/fishok/octo/pipeline)
  • this-week.md Недельный план: P1/P2/P3 задачи, блоки по проектам, гипотеза недели

3. Клиенты — изолированные проекты

Каждый клиент - отдельная папка с собственной структурой. Никакого «общего» state - изоляция предотвращает утечки контекста.

Структура папки клиента

  • CLAUDE.mdКонтекст проекта, читается автоматически
  • state.mdОперативная память, 1-2 страницы, обновляется в /day-close
  • private.mdМоя разведка, риски, гипотезы - НЕ показывать
  • shared.mdТо, что ОК показать команде клиента
  • meetings/Транскрипты созвонов
  • raw/Файлы клиента, скриншоты, выгрузки
  • work/Мини-проекты внутри клиента (сайт, CRM, презентации)

🐟 Fishok

Текущая позиция, основной доход. Тег в decisions.md: fishok

🐙 Octo Relocate

Релокация Испания. Главный путь к equity-портфелю через канал Сергея. Тег: octo

5. AI-команда

Субагенты .claude/agents/

Специализированные «роли», каждый со своим контекстом и инструментами. Изолируют сложные задачи от основного диалога.

Skills .claude/skills/

Атомарные процедуры: /intake, /day-close, /weekly-dialogue, /update-state. Вызываются по слэш-командам.

6. Правила (что держит систему от деградации)

Правило роста: новая папка или файл создаётся только когда (а) действующий перерос 200 строк, (б) появилось ≥3 однотипных артефакта, или (в) явное решение «нужно для одной из трёх целей 90 дней». Запрещено «на всякий случай».
Kill-switch: если 14 дней нет коммитов в репозитории - система признана мёртвой. 30 дней не строим следующую итерацию. Сначала ретроспектива «почему не работало».
Изоляция контекста: при работе с проектом - только его файлы. При работе с мини-проектом внутри клиента (clients/octo/work/website/) - только локальные файлы, не весь клиент. Экономия токенов и фокус.
Не читать без явного запроса: archive/ (старые версии), inbox/today.md вне /day-close, private.md клиентов в публичных артефактах.

7. Как улучшать систему

Три направления роста, в порядке приоритета:

  1. Наполнение state-файлов клиентов. Сейчас они почти пустые - заполнятся через первые /day-close и /weekly-dialogue.
  2. Новые skills под повторяющиеся операции. Если задача делается ≥3 раз - превращать в skill. Кандидаты: /dashboard-update, /client-onboarding, /decision-add.
  3. Новые субагенты под равные задачи. Кандидаты: agent для подготовки коммерческих предложений, agent для анализа звонков из meetings/.
